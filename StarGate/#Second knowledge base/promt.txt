LXC-контейнер — оптимальный вариант для Proxmox
Почему именно LXC:

Минимальные накладные расходы (буквально ~50MB RAM)
Прямой доступ к сетевым интерфейсам хоста через bridge
Быстрый старт, легко бэкапить снапшотами Proxmox
Tailscale отлично работает в привилегированных LXC
Настройка LXC для Tailscale: Контейнер должен быть привилегированным (Privileged), плюс в конфиге /etc/pve/lxc/<id>.conf добавить:

lxc.cgroup2.devices.allow: c 10:200 rwm
lxc.mount.entry: /dev/net/tun dev/net/tun none bind,create=file
Это даёт доступ к /dev/tun — без этого Tailscale не поднимет интерфейс.

Почему не другие варианты
Вариант	Проблема
Напрямую на Proxmox	Ты правильно чувствуешь — замусоривание хоста, риски обновлений, концептуально плохо
VM	Работает, но избыточно — целая виртуалка ради VPN-агента
Docker	Требует больше танцев с --net=host и /dev/tun, менее нативно

Subnet Router — самая полезная фича в твоём случае:

bash
tailscale up --advertise-routes=192.168.1.0/24 --accept-dns=false

Итого
LXC (привилегированный) + Subnet Router — это канонический способ. Лёгко, чисто, и весь хомлаб доступен из любой точки.

Да, это реализуемо! Схема называется selective exit node и это как раз сильная сторона связки Tailscale + Mihomo.

Как это работает архитектурно
Телефон (Tailscale)
       ↓
  Tailscale нода (LXC)
       ↓
   Mihomo (tproxy/TUN)
       ↙          ↘
Белый список      Всё остальное
(прямо)        (через прокси/другой exit)
Два подхода к реализации
Вариант 1 — Mihomo как шлюз поверх Exit Node
Tailscale на телефоне указывает exit node на твой LXC, трафик приходит туда, а там iptables/nftables редиректит всё в Mihomo (режим tproxy или TUN).

Mihomo рулит правилами:

yaml
rules:
  - GEOIP,RU,DIRECT        # российские IP — прямо
  - DOMAIN-SUFFIX,gosuslugi.ru,DIRECT
  - MATCH,proxied-out       # всё остальное — через прокси
Вариант 2 — Mihomo в TUN-режиме, Tailscale рядом
Mihomo поднимает свой TUN-интерфейс, Tailscale работает параллельно. Сложнее с маршрутизацией, могут конфликтовать TUN-интерфейсы.

Вариант 1 проще и надёжнее.

Практическая схема в LXC
В одном привилегированном LXC:

Tailscale — принимает трафик с телефона, объявляет себя exit node
Mihomo — слушает на tproxy порту (например 7893)
nftables/iptables — перехватывает исходящий трафик и кидает в Mihomo
Ключевые правила iptables:

bash
# Весь исходящий TCP/UDP → Mihomo tproxy
iptables -t mangle -A PREROUTING -i tailscale0 \
  -p tcp -j TPROXY --on-port 7893 --tproxy-mark 1
iptables -t mangle -A PREROUTING -i tailscale0 \
  -p udp -j TPROXY --on-port 7893 --tproxy-mark 1
Нюансы
Момент	Важно
DNS	Нужно чтобы DNS тоже шёл через Mihomo, иначе утечки. Mihomo умеет fake-ip или redir-host
Два LXC vs один	Можно разнести Tailscale и Mihomo по разным LXC — чище, легче дебажить
Tailscale exit node	В Admin Console надо разрешить Use as exit node, на телефоне выбрать эту ноду
Конфликт TUN	Если Mihomo в TUN-режиме — могут быть проблемы, лучше tproxy
Итого
Связка работает отлично, многие так и делают для "умного" выхода в интернет через домашний сервер. Самый чистый вариант — два LXC: один для Tailscale (subnet router + exit node), второй для Mihomo, и между ними обычный роутинг внутри Proxmox bridge.

Да, это DPI (Deep Packet Inspection) + активное зондирование со стороны приложений. Уже давно практикуют банки, госуслуги, некоторые стриминги.

Что именно происходит
Приложение запускается
       ↓
Пингует "маркерные" хосты (8.8.8.8, example.com, 
известные заблокированные сайты)
       ↓
Если они отвечают — приложение решает что ты за VPN
       ↓
Блокирует функционал / просит выключить VPN
Иногда проверяют не только доступность, но и latency (через VPN — выше), TTL, IP геолокацию DNS-ответов.

Решение — раздельный DNS + избирательная блокировка зондов
Mihomo это решает, но нужно правильно настроить:

1. Фейковые ответы для "маркерных" хостов
В Mihomo можно через hosts или правила возвращать для заблокированных сайтов REJECT или локальный IP — чтобы зонд не прошёл, но и ошибки соединения не было:

yaml
hosts:
  "evil-check-host.com": 127.0.0.1  # притворяемся что недоступен

rules:
  - DOMAIN,known-blocked-checker.com,REJECT
2. DNS должен быть "чистым" для белого списка
yaml
dns:
  enable: true
  fake-ip-range: 198.18.0.0/15
  nameserver:
    - 77.88.8.8  # Яндекс для RU доменов
  fallback:
    - tls://1.1.1.1  # для остальных
  fallback-filter:
    geoip: true
    geoip-code: RU
```

### 3. Геолокация IP не должна палить тебя

Если выходной узел твоего прокси имеет **RU геолокацию** — приложение счастливо. Многие используют для этого российские VPS как промежуточный hop.

---

## Более глубокая проблема — fingerprinting

Некоторые приложения смотрят на:

| Признак | Как палит |
|---|---|
| **MTU пакетов** | У VPN/tun интерфейсов часто 1280/1420 вместо 1500 |
| **Latency паттерны** | Слишком низкий пинг до "заблокированных" хостов |
| **WebRTC leak** | В браузере может светить реальный IP |
| **DNS leak** | Запросы уходят мимо твоего стека |

Mihomo с tproxy + правильный DNS закрывает большинство из этих векторов.

---

## Практический совет

Самое надёжное сейчас — это **два профиля в Mihomo**:
```
Профиль "Дома" (без exit node на телефоне):
- Mihomo только для заблокированного контента
- Белый список идёт напрямую с домашним IP
- Приложения банков видят RU IP → счастливы

Профиль "В роуминге" (с exit node):
- Весь трафик через твою коробку
- Mihomo делает вид что заблокированных сайтов нет
- Банковские приложения — отдельная история
Банковские приложения с серьёзной антифрод-защитой (Тинькофф, Сбер) могут палить VPN на уровне который практически не обходится без реального RU IP на выходе. Здесь единственное надёжное решение — российский VPS как exit hop специально для этих приложений.
